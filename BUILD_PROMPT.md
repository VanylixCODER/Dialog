# Промпт: собрать мессенджер «Dialog» с нуля

> Скопируй весь этот документ как задание агенту. Это полная спецификация: стек,
> архитектура, дизайн, фичи, схема БД, протокол реального времени и — главное —
> подводные камни WebRTC, на которых легко споткнуться.

---

## 0. Что строим (одним абзацем)

Веб-мессенджер: текстовый чат и групповые аудио/видео-звонки **с функционалом как в
Discord** (демонстрация экрана, реакции, редактирование/удаление, голосовые,
заглушка участников), **раскладкой UI/UX как в Telegram** (список чатов слева +
переписка справа, на мобиле — переключение), в **визуальном стиле «Матрицы»**
(зелёный терминал, CRT, Jarvis/Hyprland-HUD). Деплой на бесплатный хостинг,
плюс Android-обёртка, которая обновляется сама без пересборки.

## 1. Стек и архитектура

```
┌─────────────── Клиент (ванильный JS, без фреймворков) ───────────────┐
│  index.html · style.css · app.js · i18n.js · icons.js · emoji.js     │
│  matrix.js (фон-дождь) · sw.js (service worker, Web Push)            │
│  WebRTC (mesh) · Socket.IO client · MediaRecorder · WebAudio         │
└──────────────┬───────────────────────────────────────┬──────────────┘
               │ Socket.IO (WS)                         │ REST (fetch)
┌──────────────▼───────────────────────────────────────▼──────────────┐
│  Node.js + Express + Socket.IO  (server.js)                          │
│  auth.js (scrypt) · db.js (mysql2) · cache.js (ioredis, опц.)        │
└──────────────┬───────────────────────────┬──────────────────────────┘
               │ mysql2/promise            │ ioredis (опционально)
        ┌──────▼──────┐             ┌───────▼───────┐
        │  MySQL 8     │             │  Redis (кэш)  │
        │ (TiDB/Aiven) │             │  сессии+хист. │
        └─────────────┘             └───────────────┘

Внешние: Metered.ca (STUN/TURN) · GIPHY API · Web Push (VAPID)
```

**Принципы:**
- **Без фронт-фреймворков.** Чистый JS-модуль `app.js` (~1500 строк), DOM напрямую.
  Это намеренно — меньше зависимостей, легче деплой, проще отлаживать.
- **Сервер тонкий:** REST для CRUD + Socket.IO для реального времени + relay WebRTC-сигналинга.
- **Медиа (фото/видео/голос)** хранятся как data-URL (base64) в `LONGTEXT`. Лимит 20 МБ.
  Для продакшена побольше — выносить в объектное хранилище, но для старта так проще.
- **Сессии в БД** (не в памяти) — переживают рестарт процесса (важно для бесплатных
  хостингов, которые усыпляют/перезапускают сервис).
- **Redis опционален:** включается только при наличии `REDIS_URL`. Кэширует сессии и
  историю комнат. Без него всё работает, просто чуть больше запросов к MySQL.

**Технологии:** Node 20+, Express, socket.io, mysql2/promise, ioredis, web-push, dotenv.
Шрифт — JetBrains Mono. Иконки — самописные SVG (Feather-стиль, `stroke=currentColor`).

## 2. Дизайн-система «Матрица» (Matrix + Jarvis + Hyprland)

CSS-переменные:
```css
--bg:#000700; --bg-2:#020a04; --bg-3:#04120a;
--panel:rgba(0,255,90,.035); --panel-2:rgba(0,255,90,.07); --panel-3:rgba(0,255,90,.13);
--border:rgba(0,255,90,.22); --border-strong:rgba(0,255,90,.45);
--text:#3df58a; --text-dim:#138a4a; --green:#00ff5a; --green-bright:#b6ffd2;
--danger:#ff4444; --glow:0 0 8px rgba(0,255,90,.55); --glow-soft:0 0 14px rgba(0,255,90,.35);
```

Обязательные визуальные эффекты (всё на CSS/canvas, без картинок):
- **CRT-сканлайны** поверх всего: `body::before` с `repeating-linear-gradient` зелёных
  полос, `z-index:9999; pointer-events:none`.
- **Матрица-дождь** на экране входа — `<canvas>` с падающими 
  ка­таканой/символами (`matrix.js`).
- **Угловые скобки-рамки (Jarvis)** на карточках/тайлах: `::after` с 8 линейными
  градиентами по углам.
- **Скан-луч** вдоль шапки чата (бегущая горизонтальная линия с `--glow`).
- **Технологичная сетка** под чатом (две линейные сетки + radial-mask).
- **Пульсирующее кольцо** вокруг логотипа, **неоновая бегущая полоса** сверху сайдбара.
- Всё моноширинным шрифтом, `text-shadow` лёгкое зелёное свечение.
- Уважать `@media (prefers-reduced-motion: reduce)` — отключать анимации.
- Скроллбары, выделение текста, каретку — перекрасить в зелёный.

Тон: «хакерский терминал/HUD», но читабельный и не кислотный.

## 3. Раскладка UI/UX (как в Telegram)

Десктоп — CSS Grid `330px | 1fr | (инфо-панель)`:
- **Слева — список чатов** (`.chatlist`): шапка с аватаром/именем/кнопками
  (контакты, новый чат, профиль, выбор языка), поиск, список диалогов с
  последним сообщением, временем, бейджем непрочитанных, статус-точкой.
- **Справа — переписка** (`.chat`): шапка (аватар, имя, подзаголовок, индикатор
  «печатает…», кнопки: заглушить, **позвонить**, инфо), лента сообщений, композер.
- **Инфо-панель** (опц.) — участники группы / профиль собеседника.
- **Композер:** эмодзи, GIF, прикрепить файл, голосовое (зажать/тап), поле ввода
  (авто-рост), отправить.

Мобайл (`max-width:720px`): одна колонка, список ↔ чат переключаются (кнопка «назад»).
Учитывать `100dvh` + `env(safe-area-inset-*)` + `viewport-fit=cover` (хедер не должен
уезжать под адресную строку браузера).

Анимации смены чата/комнаты, появления сообщений (`msgIn`, `bubbleIn`).

**Главные экраны/модалки:** вход (вкладки логин/регистрация), новый чат (ЛС/группа),
контакты+заявки в друзья, профиль (аватар/имя/описание/статус), мини-профиль (по клику
на аватар), оверлей звонка, **полноэкранный поп-ап входящего звонка**, лайтбокс (зум фото),
панель GIF/эмодзи.

## 4. Функции

### Текст (как в Discord)
- Реалтайм-сообщения (Socket.IO), история из MySQL (кэш в Redis).
- Типы: `text`, `image`, `gif`, `video`, `audio` (голосовое), media как data-URL.
- **Реакции** эмодзи (тоггл, счётчики, подсветка своих) — хранить как JSON в колонке.
- **Редактирование** своих текстовых (инлайн, Enter — сохранить, Esc — отмена, метка
  «изменено»).
- **Удаление** своих сообщений (у всех в реалтайме).
- **@упоминания** с подсветкой и пингом (звук+уведомление).
- **Голосовые** через MediaRecorder, **фото/видео/GIF** (GIPHY-прокси на сервере),
  **эмодзи-пикер**, **лайтбокс** с зумом/перетаскиванием.
- **Тосты «печатает…»**, индикатор соединения, авто-reconnect.

### Звонки (как в Discord)
- **Mesh WebRTC** (каждый с каждым), аудио по умолчанию, камера выключена.
- **Демонстрация экрана** отдельным тайлом (как в Discord — не вместо камеры).
- **Тайлы:** видео во весь прямоугольник; камера выкл → круглый аватар по центру.
  Speaking → зелёное свечение. Демка → растянутый тайл сверху.
- **Кнопки:** микрофон (+ дропдаун: **выбор устройства ввода/вывода**, шумодав-тоггл),
  камера, демонстрация, повесить. Дропдаун у микрофона — в стиле Discord.
- **Заглушка/громкость** каждого участника отдельно.
- **Оконный режим** на ПК (перетаскиваемое мини-окно звонка).
- **Глобальный вызов:** входящий звонок долетает до юзера в любом чате (и через push).
- **Полноэкранный поп-ап входящего** с аватаром, пульсом, принять/отклонить, рингтоном
  и cava-визуализатором частот (WebAudio analyser → canvas).

### Социалка
- **Друзья:** заявки/принять/отклонить/удалить. **Блокировки** (Discord-стиль:
  сообщения скрыты, клик — показать).
- **Гейтинг ЛС:** писать можно только друзьям или участникам общей группы; иначе
  сообщение не идёт и автоматически уходит заявка в друзья.
- **Группы** (приватные, контроль доступа по членству).
- **Профиль:** аватар, имя, описание, дата регистрации. **Мини-профили** по клику.
- **Статусы:** online / dnd (не беспокоить) / invisible. Живое присутствие
  (broadcast друзьям). **DND глушит звонки, звуки и push** (на клиенте И на сервере).
- **i18n:** EN по умолчанию + RU, переключатель.

## 5. Модель данных (MySQL, utf8mb4, InnoDB)

```sql
users(login PK, name, salt, hash, avatar LONGTEXT, description, status, created_at)
messages(id PK AI, room, from_login, name, ts BIGINT, type, text, media LONGTEXT,
         media_name, reactions TEXT/*JSON {emoji:[login,...]}*/, edited TINYINT,
         KEY(room, id))
sessions(token PK, login, created_at)            -- токен-сессии, переживают рестарт
chat_groups(id PK AI, name, owner, created_at)
group_members(group_id, login, PRIMARY KEY(group_id, login))
relations(login, target, type/*friend|block|request*/, PRIMARY KEY(login,target,type))
push_subs(endpoint PK, login, sub TEXT)          -- Web Push подписки
```

Ключи комнат: ЛС — `@dm:<loginA>~<loginB>` (логины отсортированы), группа — `@grp:<id>`.
`initSchema()` создаёт таблицы идемпотентно (`CREATE TABLE IF NOT EXISTS`) + миграции
колонок через `try { ALTER TABLE ... } catch {}`.

**Подключение к БД (важный урок):** поддержать И единый `DATABASE_URL`, И отдельные
`DB_HOST/DB_PORT/DB_USER/DB_PASS/DB_NAME` — у managed-баз (TiDB) пароли со спецсимволами
ломают URL-парсинг. Managed требует TLS: `DB_SSL=true` →
`ssl:{minVersion:"TLSv1.2", rejectUnauthorized:false}`.

## 6. Протокол реального времени (Socket.IO)

REST: `/api/register|login|me|profile|profile/:login|avatar/:login|user/:login|
groups|groups/:id/leave|relations|friend|presence|gif|ice|push/key|push/subscribe`.

Socket события:
- `identify {token}` — привязать сокет к юзеру (ловить звонки/пинги где угодно).
- `join {room, token}` — войти в комнату (контроль доступа к @dm/@grp!), отдать историю.
- `message` / `typing` / `msg-delete` / `msg-edit` / `msg-react`.
- `signal {to, kind, data}` — relay WebRTC (адресно по socketId).
- `media {to?, kind:"cam"|"screen", on}` — явное состояние камеры/экрана (см. §7).
- `call-invite {title}` → `call-invite` (в комнате) + `call-ring` (глобально членам).
- `set-status` — присутствие.

Сервер шлёт: `history`, `message`, `msg-deleted/edited/reaction`, `dm-ping`,
`call-invite/call-ring`, `peer-joined/left`, `presence`, `dm-blocked`, `relations-changed`.

## 7. WebRTC — КРИТИЧЕСКИЕ УРОКИ (читать до написания звонков!)

Это место съело больше всего итераций. Делай сразу правильно:

1. **Детерминированный инициатор, НЕ perfect-negotiation с glare.**
   Когда оба участника одновременно создают offer (glare), после неявного rollback
   ICE-кандидаты генерируются со «старым» ufrag и отбрасываются другой стороной — связь
   падает в `ICE failed`, хотя relay-кандидаты вроде бы есть. Решение: offer создаёт
   ТОЛЬКО тот, у кого `socket.id < peerId`. Второй всегда отвечает. Никаких гонок.
   Для поздней ренеготиации не-инициатор шлёт `need-offer`, инициатор переофферивает.

2. **Предсоздавай трансиверы, переключай через `replaceTrack` — без ренеготиации.**
   В `ensurePeer` сразу создай 3 sendrecv-трансивера: аудио (микрофон), видео-камера,
   видео-экран. Включение камеры/демки = `sender.replaceTrack(track)` (и `null` чтобы
   выключить). Это полностью убирает позднюю ренеготиацию и связанные с ней гонки.
   Маппинг входящих треков — по `e.transceiver === st.camTx/screenTx/audioTx`.

3. **Показ камеры/экрана — по ЯВНОМУ сигналу `media`, а не по `track.onunmute`.**
   `onunmute` при `replaceTrack` на заранее созданном трансивере срабатывает ненадёжно.
   Поэтому при включении камеры/демки шли socket-событие `media {kind, on}` — у
   собеседника по нему показывай/прячь видео/тайл. При подключении нового пира — синкай
   ему своё текущее состояние адресно.

4. **TURN обязателен.** Только STUN не пробивает симметричный NAT (мобильные сети).
   Бери бесплатный Metered.ca (`METERED_API_KEY`) — сервер дёргает их API и отдаёт
   `iceServers` клиенту через `/api/ice`. Дай выбрать STUN + TURN(udp) + TURNS(tcp/443).
   Для диагностики — режим `?relay` (форс `iceTransportPolicy:"relay"`) и лог типов
   кандидатов (есть ли `typ relay` исходящие И входящие).

5. **Буферизуй ICE-кандидаты** до `setRemoteDescription` (иначе `No remoteDescription`).
   ICE-restart при `failed` (через инициатора), с лимитом попыток (не зациклить).

6. **Аудио:** ремоут играет ЧЕРЕЗ сам `<video>`/`<audio>` элемент (`muted=false` +
   `play()`), НЕ через WebAudio `createMediaStreamSource` (в Chrome даёт тишину на
   ремоут-стримах). Autoplay-политика мобилок: фоллбэк `play()` по первому клику.

7. **Оптимизация трафика (важно для бесплатного TURN, 500 МБ/мес):** аудио-only старт
   (камера не запрашивается), низкое видео 640×360@20–24fps, лимиты битрейта через
   `sender.setParameters` (аудио ~40к, видео ~300к, демка ~1.8М). Демка 30fps.

8. **Мобильная вёрстка звонка:** НЕ grid с фиксированным `aspect-ratio` (схлопывает
   тайлы) — а вертикальный скроллящийся flex-стек, тайлы во всю ширину, демка сверху.

9. **Звонок не должен падать при сворачивании (для мобилки/TWA):**
   - `MediaSession` (metadata + `playbackState:"playing"`) — ОС держит вкладку живой и
     показывает на локскрине, как медиа-плеер.
   - `Wake Lock` во время звонка (пере-запрашивать на `visibilitychange`).
   - Тихий keep-alive (WebAudio осциллятор с почти нулевым gain) против заморозки вкладки.
   - На `visibilitychange→visible` — `play()` всем видео (мобилки ставят на паузу в фоне).

## 8. Уведомления, PWA, Android-обёртка

- **Web Push (VAPID):** `web-push` на сервере, `sw.js` на клиенте, таблица `push_subs`.
  Эндпоинты `/api/push/key` и `/api/push/subscribe`. Шли push при сообщении (тем, кто не
  в этой комнате) и при звонке. **DND-юзерам push не слать.**
- **Service worker:** показывает уведомление; для звонка — `requireInteraction`, вибрация,
  экшены «Принять/Отклонить»; по клику открывает нужный чат (`?room=...&autojoin=1`),
  при открытом фокусе приложения — не дублировать (in-app поп-ап сам покажет).
- **PWA-манифест** (`manifest.json`) + иконка (SVG ok) + `theme-color` + apple-touch-icon
  → ставится на домашний экран; на iOS push только для установленной PWA.
- **Android = TWA** (Trusted Web Activity, НЕ голый WebView — тот не умеет Web Push!).
  Минимальный Gradle-проект с `com.google.androidbrowserhelper`, грузит URL сайта →
  **обновления прилетают сами, APK пересобирать не надо.** Подтверждение домена через
  `/.well-known/assetlinks.json` (express не отдаёт dotfiles по умолчанию — добавить роут),
  отпечаток ключа подписи в assetlinks → без адресной строки. Делегация уведомлений
  (`DelegationService`) — Web Push показывается как уведомление приложения. Адаптивная
  векторная иконка (minSdk 26 → без PNG).

## 9. Безопасность

- Пароли — **scrypt** + случайная соль, сравнение `timingSafeEqual`.
- Токен-сессии в БД (+ кэш Redis), `Authorization: Bearer`.
- **Контроль доступа в `join`:** к `@dm:` пускать только две стороны, к `@grp:` — только
  членов (проверять в обработчике, не доверять клиенту).
- Имя сообщения брать из профиля на сервере (клиент не подделает).
- Гейтинг ЛС (см. §4). Эскейпить HTML в сообщениях (XSS).
- Лимит payload Socket.IO (`maxHttpBufferSize`), лимит файлов 20 МБ.
- Секреты только в env (`.env` в `.gitignore`): `DATABASE_URL`/`DB_*`, `REDIS_URL`,
  `METERED_API_KEY`, `GIPHY_KEY`, `VAPID_PUBLIC/PRIVATE/SUBJECT`, `TURN_*`.

## 10. Деплой

- **Хостинг:** Render (бесплатный web-service, Docker, регион Frankfurt). `Dockerfile`
  на `node:22-alpine`, `npm ci --omit=dev`. Учесть, что free-tier усыпляет сервис —
  Socket.IO авто-reconnect + сессии в БД спасают.
- **БД:** TiDB Cloud Serverless (постоянный бесплатный тариф, MySQL-совместимый, НЕ
  истекает — в отличие от триалов Aiven). `DB_SSL=true`.
- **Локально:** docker-compose (MySQL 8 + Redis 7), HTTPS по самоподписанному серту
  (камера/экран требуют secure context по сети).

## 11. Чек-лист «не наступить на грабли»
- [ ] WebRTC: детерминированный инициатор + предсозданные трансиверы + `media`-сигнал (§7).
- [ ] TURN настроен и проверен режимом `?relay`.
- [ ] Ремоут-аудио играет через media-элемент, не через WebAudio source.
- [ ] БД: и `DATABASE_URL`, и отдельные `DB_*`; `DB_SSL` для managed.
- [ ] Сессии в БД, не в памяти.
- [ ] Мобильный звонок: flex-стек, keep-alive (MediaSession/WakeLock/visibility).
- [ ] Android: именно TWA + assetlinks (роут для `.well-known`), не голый WebView.
- [ ] DND глушит и на клиенте, и на сервере (push).
- [ ] `100dvh` + safe-area на мобиле.
- [ ] Эскейп HTML, контроль доступа к комнатам, секреты в env.
